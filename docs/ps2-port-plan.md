# PS2 Port Plan (Demake)

This is a demake, not a port. The plan is organized around two hard limits: **32 MB main RAM** and **4 MB VRAM (GS local memory)**. Everything else is negotiable; those two are not.

This work lives on `ps2-port`, branched from `android-macos-iphone-windows64-vibecode-test`, so it inherits the ILP32/LP64 fixes, `time_compat`, `RTS_GAMEMEMORY_ENABLE` support, and headless/FileSystem/FramePacer work already done there. It does not target `main` directly.

## Ifdef discipline

Per repo convention, **every PS2-specific change must be guarded**, not just visually distinct. Concretely:

- Guard macro: `__PS2__` (defined explicitly by `cmake/toolchains/ps2.cmake`, matching the ps2sdk/gsKit convention -- ps2sdk does not predefine it on its own) wrapping all PS2-only code paths (VU1 transform, GS register packets, IOP/audsrv audio, libpad input, PS2 memory arena).
- Any change that looks like a general correctness fix but is being made *for* PS2 must still go behind the ifdef. Do not touch shared files (`DX8Wrapper`, `BgfxBackend.cpp`, `FramePacer.cpp`, `GameMemory`, cmake) without a guard, even if the fix looks universally right — assume Windows/Linux/macOS/Android/iOS/Switch/WASM all currently work and must not regress.
- If a change *would* legitimately benefit other targets (e.g. a genuine bug found while porting, or a shared budgeting utility), do **not** fold it into this branch's PS2 ifdef. Instead:
  1. Land it as its own guarded or platform-neutral change, ideally cherry-picked/PR'd back onto `android-macos-iphone-windows64-vibecode-test` (or `main` if it's a real bug) independently of the PS2 work.
  2. Only add it to the "shared uplift" list below once it's landed upstream of this branch, so PS2 consumes it rather than forking it.
- New PS2-only files (VU1 microprograms, GS backend, asset repack tool) don't need ifdefs internally, but their CMake inclusion must be gated on the PS2 target so other platform builds never see them.

### Shared uplift candidates (land separately, not inside PS2 ifdefs)
Track here as they're identified during the port; move to "landed" once merged elsewhere.
- INI/template lazy-hydration (if TheThingFactory eager-load becomes a real bottleneck) — likely helps Android low-RAM devices too.
- Any additional ILP32 fixes discovered (should be rare — wasm32 already exercised this).
- Controller/UI-scaling primitives from Phase 4 below, if this experiment graduates — shared with a future Switch effort.

## Hardware constraints

| Resource | PS2 | This engine currently wants |
|---|---|---|
| CPU | 294 MHz MIPS R5900 (EE), in-order, non-IEEE FPU | 800 MHz+ PIII-class minimum historically |
| Main RAM | 32 MB | ~250–400 MB on current ports |
| VRAM (GS) | 4 MB, no shaders, fixed-function, huge flat-fill rate | ~100+ MB of TGA/DDS assets |
| Audio RAM | 2 MB (SPU2), IOP co-processor | ~600 MB music/voice/SFX |
| Disc | 4.7 GB DVD, slow seeks | 3.4 GB install |

The EE is little-endian ILP32 — same class as wasm32, already proven to run this codebase. The GS being fixed-function is actually a closer shape to the original DX8 fixed-function path than bgfx.

## Philosophy: keep systems first, cut only on measured evidence

Earlier drafts of this plan cut video, online networking, and retail/replay determinism on day one as assumptions ("PS2 obviously can't do that"). That assumption was wrong on all three counts, and the corrected rule going forward is: **keep a system until a real measurement (RAM budget, CPU profile, or code-size report) proves it doesn't fit**, not because it sounds implausible for 2000s console hardware. PS2 homebrew has working prior art for the first two, and a live in-project effort addresses the third:

- **Video is not a cut.** [Simple Media System (SMS)](https://github.com/ps2homebrew/SMS) is a real, shipped PS2 homebrew media player built on a heavily modified FFmpeg fork (DivX/XviD/MPEG-2 decode up to 1024x1024, MP3 audio), hand-tuned to use the EE's IPU hardware MPEG2 decoder and VU cycles instead of brute-force scalar decode. Cutscene/video playback is a real Phase 6+ candidate using that fork (or techniques from it) as a base, not an assumed impossibility. Budget it against IOP/EE cycles and SPU2 RAM once profiled, rather than deleting it up front.
- **Online networking is not a cut.** ps2sdk ships `smap.irx` (driver for the official PS2 Network Adapter's Ethernet chip) and `ps2ip.irx` (lwIP-based TCP/IP stack) as standard, working ports/libraries — this is real, supported hardware (SCPH-10281, and built-in Ethernet on later fat models), not a stretch. The GameSpy master-server integration specifically is dead everywhere (the service no longer exists, PS2 or otherwise) and can be dropped as dead code independent of the port, but the LAN/socket-level GameNetwork code (direct-IP play, broadcast discovery) should be kept and ported against ps2ip's BSD-socket-like API, then cut only if a real profiling pass shows it doesn't fit the RAM/CPU budget.
- **Retail/replay determinism is not a cut either.** A software FPU mode targeting cross-platform determinism (denormals + rounding-mode behavior matching x86) is in active development elsewhere in the project. If it lands, the R5900's non-IEEE hardware FPU stops being the blocker — the port would run sim math through the software path instead of hardware FPU instructions. Don't foreclose this in PS2-specific code: keep sim-side floating point behind the same abstraction that software FPU mode will hook, and revisit "permanent fork target" status once that effort has a landing point to build against, rather than assuming divergence is inherent to the hardware.

Keep this "measure, then cut" discipline for everything below the two big-ticket items too. The systems most likely to still need trimming after real measurement (informed by console generation and genre norms, not certainty) are: shadow volumes/projected shadows, the water reflection/soft-edge pass, smudge, heat effects, and particle counts — these are the fringe rendering systems the Android/water-tracks work already showed are disproportionately expensive relative to their visual contribution. Treat them as the first profiling targets in Phase 3, not as pre-decided cuts.

## Known-impossible or clearly-dead (actual cut list)

Unlike the systems above, these are not measurement-dependent — they're already dead on every platform this codebase targets, independent of PS2:

1. **GameSpy master-server integration specifically** (not the LAN networking code around it — see above). The service is defunct on every platform this codebase targets; this is general dead-code removal, not a PS2-specific sacrifice.
2. **All dev tooling, Tracy, debug builds.** MFC/Win32/DirectX-based tools obviously don't cross-compile for a freestanding MIPS target (already disabled in `cmake/toolchains/ps2.cmake` via `RTS_BUILD_*_TOOLS OFF`).

Retail/replay determinism is deliberately *not* in this list — see the software-FPU note above.

## Likely tuning knobs (confirm with real numbers before touching)

These are informed guesses about what Phase 1–3 profiling will probably force, not decisions:

- Music/EVA lines may need to move from fully-resident assets to IOP-streamed ADPCM if SPU2's 2 MB can't hold them resident — confirm against actual asset size once the audio pipeline is profiled, don't pre-cut.
- Particle counts, shadow techniques, and water effects are likely tuning targets (see above) — cut only the specific technique profiling flags as expensive, not particles/shadows/water wholesale.
- Player/map size caps will be set by the Phase 2 pathfinding/AI CPU profile, not guessed in advance.
- Single-language localization and campaign scope (Zero Hour-only vs. both) are asset-size decisions to make once the actual `.big` footprint is measured against the disc/RAM budget, not assumed.
- `RTS_GAMEMEMORY_ENABLE` may need `OFF` on PS2 the way Switch needed it (see `cmake/toolchains/ps2.cmake` note), but only flip it if the custom pool allocator is actually observed to misbehave under this GCC/newlib combination — don't disable it preemptively.

## Rendering: flat/Gouraud as the foundation, not a fallback

The GS fill-rates untextured Gouraud triangles extremely fast; its weakness is textures (4 MB VRAM, constant DMA streaming). Invert the usual priority:

**Tier 0 — flat/Gouraud (first playable target):**
- Units: no textures, per-faction/house vertex color, Gouraud-lit from existing vertex normals. This is the N64-style look, and on this hardware it's cheap and correct, not a compromise.
- Terrain: bake blended tile textures down to per-vertex color at map load (sample source tiles at each grid vertex, average) — one Gouraud mesh, near-zero VRAM.
- Water: flat translucent quad, alpha blend (GS alpha blending is free).
- UI: the one place textures stay. Control bar and cursor must remain readable — keep a persistent ~1 MB VRAM slot for UI atlases, palettized (see Tier 1).

**Tier 1 — selective palettized textures (stretch, after Tier 0 ships):**
- GS supports native 4-bit/8-bit CLUT textures. Offline pipeline quantizes a whitelist (terrain macro texture, a handful of hero unit skins) to 8-bit 64×64/128×128 CLUT (4–16 KB each, streamable). Everything else stays flat-shaded.

**Backend integration point:** not bgfx — bgfx has no GS backend and its shader-centric model doesn't fit fixed-function GS. Implement at the `DX8Wrapper` seam instead (`Set_Texture`, render state, `Draw_Indexed_Primitive` map naturally onto GS register packets — the same seam the original DX8 device already fills). Bring-up with **gsKit** first (CPU-transformed, slow but simple), then move vertex transform to a **VU1 microprogram** once Tier 0 is functionally correct. The VU1 program is the single largest chunk of new PS2-specific code and needs specialist time; there is no fallback if it stalls (see risks).

## Memory budget (32 MB, defended from day one — treat as the project's controlling document)

- Code + static data (`-Os`, `--gc-sections`): **10 MB** — measure in Phase 1 with the *full* feature set (networking and video included) before cutting anything; if the ELF alone eats 20 MB, that's the signal to start trimming, not an assumption to build in up front.
- Object/INI templates (lazy-load if measurement shows eager-load is too big): **4 MB**
- Map + pathfinding grids: **4 MB**
- Game objects/partition/AI: **5 MB** (unit cap set by Phase 2 profiling, not guessed)
- Baked terrain vertex mesh + W3D geometry (lowest LOD only, shipped on disc): **5 MB**
- Streaming/IO buffers, audio staging, network (ps2ip/smap) buffers, slack: **4 MB**

The PC-side asset repack tool is a real subproject: reads `.big` files, strips to lowest LODs, quantizes whitelisted textures, converts audio to ADPCM as needed, and emits a seek-friendly PS2 disc image. Scope of what it strips follows the measured tuning knobs above, not a fixed list decided today.

## Toolchain setup (done, reproducible)

The ps2dev/ps2sdk toolchain is installed and build-verified as of this writing:

- Installed to `C:/code/ps2dev` from the official Windows release (`ps2dev/ps2dev` GitHub releases, `ps2dev-windows-latest.tar.gz`), matching the existing `C:/code/devkitPro` convention for the Switch toolchain.
- **Known issue (Windows release only):** the `mips64r5900el-ps2-elf-*` binaries are 32-bit (x86) and the release tarball does not bundle their MinGW32 runtime DLLs. Symptom is a silent exit code with no diagnostic text (or, under Git Bash, a garbled `error while loading shared libraries: ?: cannot open shared object file`). Fixed by sourcing `libiconv-2.dll`, `libwinpthread-1.dll`, `libcharset-1.dll`, `libzstd.dll`, `libgcc_s_dw2-1.dll`, `libstdc++-6.dll`, `libatomic-1.dll`, `libgomp-1.dll`, `libquadmath-0.dll`, `libgmp-10.dll`, `libgmpxx-4.dll`, `libisl-23.dll`, `libmpc-3.dll`, `libmpfr-6.dll` from the official MSYS2 `mingw32` package repo (`mirror.msys2.org/mingw/mingw32/`) and copying them into **all four** directories that hold a toolchain binary GCC actually invokes: `ee/bin`, `ee/libexec/gcc/mips64r5900el-ps2-elf/15.2.0`, `ee/mips64r5900el-ps2-elf/bin` (the sysroot-relative `as`/`ld`/`ar` GCC's driver resolves at link time — easy to miss), and the `iop/` equivalents of the last two.
- `mips64r5900el-ps2-elf-gcc-ar`/`-gcc-ranlib` wrappers crash outright even with the DLLs present (unrelated cause, not chased down) — use plain `mips64r5900el-ps2-elf-ar`/`-ranlib` instead. Already reflected in `cmake/toolchains/ps2.cmake`.
- CMake gotcha: `target_link_libraries()` treats the bare word `debug` as a reserved build-configuration keyword (like `optimized`/`general`), not a literal library name — since ps2sdk's console library is itself named `libdebug.a`, this silently drops it. Use `-ldebug` (flag form) instead.
- `cmake/toolchains/ps2.cmake` encodes all of the above; a standalone smoke-test target lives at `ps2-port/bringup/` (not part of the main engine build) and has been built and run successfully in PCSX2, confirming the full compile → link → boot pipeline works before any engine code is touched.
- PCSX2 (2.6.3) is set up with a real BIOS dump and `[EmuCore/CPU] ExtraMemory = true` in `PCSX2.ini` (Settings > Advanced > "Enable 128MB RAM (Dev Console)" in the UI) — this is real devkit hardware behavior PCSX2 emulates, not a hack, and gives headroom to prototype against before narrowing to the retail 32 MB budget. Confirmed via emulog: `128MB RAM is enabled`. Boot a built ELF directly with `pcsx2-qt.exe -batch -elf <path> -fastboot`.
- **SDL3 has genuine, unpatched upstream PS2 support** (video, audio, joystick/libpad, filesystem, threads, timer — plus its own GS-based 2D render backend), unlike Switch which needed a community fork/patch. `cmake/toolchains/ps2.cmake` sets the bare `PS2`/`PLATFORM_PS2`/`EE` CMake variables ps2dev's own toolchain convention (and SDL3's CMakeLists `elseif(PS2)` branch) expect; `cmake/sdl3.cmake` now treats PS2 like the other static-link cross targets (Switch/iOS/Emscripten). Build-verified at `ps2-port/bringup-sdl3/`: SDL3 configures with `Enabled backends: Video drivers: dummy offscreen ps2 / Render drivers: ps2 / Audio drivers: disk dummy ps2 / Joystick drivers: ps2 virtual`, compiles fully, links, and `SDL_Init(SDL_INIT_VIDEO)` succeeds booted in PCSX2. This means Phase 4/5 should wire the existing `SAGE_USE_SDL3` GeneralsMD backend to PS2, not write bespoke libpad/audsrv glue from scratch.

## Real engine build status: FIRST FULL COMPILE + LINK ACHIEVED

The `ps2-generalsmd` CMake preset configures, compiles, and **links the complete, real `generalszh` executable** (195 MB ELF, `GeneralsMD/Release/generalszh`) against `GGC_RENDER_BACKEND=ps2` and `SAGE_USE_SDL3=ON` — all 1278 objects, zero errors, confirmed reproducible from a fully clean `rm -rf build && cmake --preset ... && cmake --build ...`. This is the actual Zero Hour engine, not a smoke test; `readelf -h` confirms a valid `EXEC` MIPS/5900 ELF. Nothing has been run on hardware/PCSX2 yet — this is compile+link only, no data files, no runtime validation. Fixes made to get here, in the order hit:

- **New `GGC_RENDER_BACKEND=ps2` value** (`cmake/render-backend.cmake`, `Core/Libraries/Source/WWVegas/WW3D2/RenderBackend.cpp`, new `PS2Backend.h`/`.cpp`). `PS2Backend : public DX8Backend` is a Tier 0 stub (does not submit draws yet) — `IRenderBackend.h` turned out to have only **one** pure-virtual method (`Get_Ambient()`), everything else already defaults to a no-op, so the stub itself is tiny. Selecting `ps2` forces `GGC_BGFX_STANDALONE=ON` automatically, reusing the exact DX8Wrapper-stub-device infrastructure (`StubD3D8Device`, `D3DXStandaloneStubs`) Android/Linux/macOS/iOS already rely on for their bgfx-standalone builds — no bgfx library involved, no shader compilation.
- **Miles/Bink SHARED library**: both stub SDKs hardcode `add_library(... SHARED ...)`; CMake refuses this for a target with no dynamic linker. Already had an iOS/Emscripten precedent for building them STATIC instead — extended to PS2 (`cmake/miles.cmake`, `cmake/bink.cmake`).
- **`compat/win32_shims/windows.h`**: `#include <dlfcn.h>` and the real `LoadLibrary`/`GetProcAddress`/`FreeLibrary` (`dlopen`/`dlsym`/`dlclose`) implementations were already guarded `#ifndef __SWITCH__` (Switch has no dlfcn either) — extended to `__PS2__`. Also added a missing `ReleaseMutex` stub (general gap, not PS2-specific — `CreateMutex`/`CloseHandle` already had one).
- **`WWLib/stringex.h`**: `wcslcat`/`wcslcpy` inline C++ definitions conflict with ps2sdk's own C-linkage declarations in `<wchar.h>` — already guarded `#ifndef __SWITCH__` for the same reason (libnx provides them too); extended to `__PS2__`.
- **`Dependencies/Utility/Utility/thread_compat.h`**: `pthread_self()` returns an opaque pointer on ps2sdk (same as Switch's libnx), not an integer — already had a `__SWITCH__` branch using `reinterpret_cast<uintptr_t>`; extended to `__PS2__`.
- **`cmake/toolchains/ps2.cmake` global defines added**, each with a specific reason: `-D_DEFAULT_SOURCE` (the repo's `-std=c++20`, strict ISO, sets `__STRICT_ANSI__`, which hides newlib's `usleep`/`strcasecmp`/other BSD-visible declarations by default); `-D_UNIX` (several engine files, e.g. `WWLib/thread.cpp`, already have a correct pthread-based implementation gated behind `#ifdef _UNIX` with zero CMake file in this repo currently defining it for *any* platform — PS2 is the first to turn it on, and it is both safe and semantically correct since ps2sdk has real pthreads). Also removed an initial, unwarranted `-fno-exceptions -fno-rtti` — the engine genuinely throws (e.g. FileSystem INI parsing) and no other platform disables exceptions.
- **`Dependencies/Utility/Utility/intrin_compat.h`**: added a real `__PS2__` branch for `_rdtsc()` reading the R5900's COP0 Count register (`mfc0 $9`) via inline asm, alongside the existing aarch64/`__builtin_readcyclecounter` branches.
- **`WWDebug/wwdebug.cpp`**: `FormatMessage`/`FORMAT_MESSAGE_FROM_SYSTEM` (a `win32_shims/file_compat.h` stub) wasn't visible because this file's own `#ifdef _WIN32 #include <windows.h> #else #include <errno.h> #endif` never included the shim header for non-Windows — apparently already relying on being pulled in transitively via another library's PCH elsewhere. Added an explicit `__PS2__` branch that includes it directly rather than touching the shared PCH composition.
- **`WWLib/cpudetect.h`/`cpudetect.cpp`**: `sint64` typedef and a `timezone` (POSIX global, ps2sdk calls it `_timezone` like Switch's libnx) both had existing `_UNIX`/`__SWITCH__` branches to extend for `__PS2__`.
- **GameSpySDK vendored source** (fetched via `cmake/gamespy.cmake`, not our code): its own platform detection does `#ifdef __mips64` (a real GCC-predefined macro for this target triple, not something we set) and enters a native-PS2 branch demanding ancient `EENET`/`INSOCK`/`eekernel.h` headers that don't exist in modern ps2sdk. Reused the exact same `_UNIX`/`__linux__` fallback already used for Switch/Emscripten (`cmake/gamespy.cmake`). `-U__mips64` ended up promoted to the *global* PS2 toolchain flags (`cmake/toolchains/ps2.cmake`), not just the `gsinterface` target — our own engine code (`GameNetwork/GameSpy/PeerDefs.h`) also includes GameSpy headers directly and hit the identical `#error` outside that target's reach. Also patched `gssha1.h`'s redundant `uint32_t`/`uint8_t` typedef (conflicts with ps2sdk's real `<stdint.h>`) via a new `cmake/patch_gamespy_ps2.cmake` FetchContent `PATCH_COMMAND`, mirroring the existing `patch_miles_static.cmake` pattern.
- **Real `ioctl()` bridge** (new `compat/win32_shims/ps2_ioctl_compat.cpp` + `ps2_ioctl_decl.h`): ps2sdk's own `sys/socket.h` `#define`s `ioctlsocket`/`lwip_ioctl` in terms of a plain variadic `ioctl(fd, request, ...)` that **no ps2sdk library actually exports** — only the fixed-arity `_ps2sdk_ioctl(int, int, void*)`. Wrote a real forwarding function (not a stub) bridging the two, force-included into GameSpySDK's C translation units via `-include` on `gsinterface` (they call `ioctl()` directly and don't see our compat headers) and included directly by `winsock.h` for our own socket code.
- **`Int`/`UnsignedInt` typedef mismatch, fixed at the root** (`Core/Libraries/Include/Lib/BaseTypeCore.h`): ps2sdk's newlib (MIPS n32 ABI) defines `int32_t`/`uint32_t` as aliases of `long`/`unsigned long`, not `int`/`unsigned int` — same width, different C++ type, so any of the countless functions across this ~2M-line codebase taking a plain `int*` rejected an `Int*` argument ("invalid conversion"). Redefining `Int`/`UnsignedInt` as the literal built-in types on `__PS2__` fixed the whole class of errors at once instead of casting every call site.
- **`.c` vs `.cpp` PCH gotcha**: adding a plain C source file to a C++-only static library target (`core_wwlib`) made CMake generate a *C-language* precompiled header for the whole target, which then tried to compile the library's own C++-only headers (`class`, `template`) as C and failed outright with bizarre-looking cascading errors. Fix: make new PS2 compat sources `.cpp` with `extern "C"` linkage, never `.c`, in a target that's otherwise pure C++.
- **`endian_compat.h`**: had a real `#error platform not supported` for any platform not in its explicit list. ps2sdk ships `<sys/endian.h>` with real `htobe16`-family macros, identical in shape to Switch's — extended the existing `__SWITCH__` branch to `__PS2__` (two spots: the macro-providing `#if` and the `SwapType16/32/64` typedef `#if`).
- **`__declspec` unhandled by GCC**: Miles/Bink's stub headers use bare `__declspec(dllimport/dllexport)` with no non-Windows guard. Clang (Android/Linux/macOS/iOS) accepts `__declspec` unconditionally as a compatibility extension; GCC does not, and confirmed `-fms-extensions` does *not* add this (that's a mingw/PE-target-only GCC feature, not a general frontend flag). Defined `__declspec(x)=` as a no-op function-like macro globally instead — the standard portable fix.
- **Missing `win.h` include in two WWAudio headers** (`SoundBuffer.h`, `Utils.h`): both use `HANDLE`/`CRITICAL_SECTION`/`LPCTSTR` directly, relying on some other translation unit's PCH to have pulled in `win.h` first — which didn't happen for these specific PS2 library compiles. Added the include directly rather than chasing PCH composition per-library.
- **stb_truetype.h missing** (`WW3D2/CMakeLists.txt`): `render2dsentence.cpp` needs it on any non-Windows backend (no GDI), normally supplied by bgfx's vendored copy — but PS2 never fetches bgfx. Added a standalone, lightweight `FetchContent` of upstream `nothings/stb` directly for the `ps2` backend value, forced into a `stb/` subfolder so the existing `#include "stb/stb_truetype.h"` resolves unchanged.
- **Link-time, not compile-time, from here**: `-latomic` needed for `__atomic_fetch_add_4`/`__atomic_test_and_set`/etc. (GCC emits real function calls for `std::atomic<T>` ops the R5900 has no hardware instruction for) — ps2dev's toolchain *does* ship a real `libatomic.a`, just not linked by default. Had to go through `CMAKE_C/CXX_STANDARD_LIBRARIES` (appended after all objects/libraries on the link line) rather than `CMAKE_EXE_LINKER_FLAGS` (placed before them) — static-library symbol resolution is left-to-right, so `-latomic` before the `.a` files referencing `__atomic_*` never resolved anything.
- **`__atomic_test_and_set` (unsuffixed) doesn't exist** in this toolchain's `libatomic.a` at all (confirmed via `nm` — only the size-suffixed `_1/_2/_4/_8/_16` forms are defined) even though libstdc++'s `atomic_flag::test_and_set()` calls the generic name directly on this MIPS target. Since `atomic_flag` is always exactly 1 byte, added a real forwarding definition to `__atomic_test_and_set_1` (not a stub — this restores correct atomic semantics, just via the differently-named implementation that actually exists). Signature must match GCC's builtin exactly (`volatile void*, int`) or the compiler rejects the "ambiguates built-in declaration."
- **`gethostname()`** has no ps2sdk implementation at all (no DNS-style hostname concept on an IP-configured console). Added a Tier 0 stub (returns the fixed string `"ps2"`) in the same new compat file — needed by `GameNetwork/IPEnumeration.cpp`, `WWDownload/FTP.cpp`, and GameSpySDK's socket code, none of which need a *real* hostname to link and run today.

## Runtime boot debugging (in progress)

First real boot attempts with actual retail game data (`.big` files etc.) copied next to the ELF, via PCSX2's `host:` filesystem. Progress is measured in "how many seconds of real engine execution before it resets to the BIOS system menu (`rom0:OSDSYS`)" — a clean reset like this (not a raw TLB-miss/garbage-instruction crash) generally means the program actually exited or hit an unhandled condition, not that it jumped into corrupted memory.

- **Real crash #1 (fixed): `-Wl,--gc-sections` corrupts `.eh_frame` on this toolchain.** First full boot executed real VU math correctly, then crashed with a stream of `TLB Miss`/`Unrecognized op` errors — the CPU jumping into garbage and executing it as code. Traced with a minimal standalone repro (`ps2-port/bringup-exceptions`): a plain cross-function `throw`/`catch` crashed identically, inside libgcc's `get_cie_encoding` (DWARF CFI parser) reading a garbage CIE augmentation string, then inside `__deregister_frame_info_bases` walking a corrupted frame list. Removing `-Wl,--gc-sections` from the link line (in `cmake/toolchains/ps2.cmake`) made the identical repro run clean with zero TLB misses, and the crash class disappeared entirely from the full engine boot too. GC'ing code sections without fully preserving CIE/FDE linkage between them is a known class of bug on less-common ld/target combinations; not re-adding this without a demonstrated need and a verified-safe fix.
  - Ruled out first (dead end, reverted): disabling `RTS_GAMEMEMORY_ENABLE` (matching Switch's real memory-pool-corruption fix) did NOT change this symptom. Kept as a `--gc-sections`-only fix instead of stacking two speculative changes; GameMemory is back to its default (ON).
- **Real bug #2 (fixed): `std::filesystem::create_directories()` always returns `EPERM` on this ps2sdk/newlib/libstdc++ combination, even when the directory is actually created.** After the crash fix, boot progressed to `GlobalData::init()` and immediately hit `IOPHLE mkdir_HLE failed for ''` (empty path) — but disassembling the real binary and adding a temporary `host:`-file diagnostic proved `m_userDataDir` itself was correct (`.\Command and Conquer Generals Zero Hour Data\`, non-empty) at that point, so the empty-path mkdir was a red herring from a *different*, still-unlocated call, not `GlobalData::BuildUserDataPathFromRegistry()`. Chasing the real `StdLocalFileSystem::createDirectory()` (the function `SDL3GameEngine::createLocalFileSystem()` actually wires up — not `Win32LocalFileSystem`) showed it uses `std::filesystem::create_directories()`, not the `mkdir()` shim directly. A minimal repro (three path variants: `./Foo/`, `Foo/`, `host:Foo/`) proved `std::filesystem::create_directories()` returns `ec=EPERM` for all of them, while a plain `mkdir()` on the *identical* path succeeds every time immediately before/after. Fixed by replacing `std::filesystem::create_directories()` with a manual component-by-component `mkdir()` loop under `__PS2__` in `StdLocalFileSystem.cpp` (tolerating `EEXIST`), keeping the `std::filesystem` path for every other platform unchanged.
  - Side lesson: PCSX2's IOP HLE only logs `mkdir_HLE`/`remove_HLE` on *failure* — successful calls are silent. Don't assume "no log line" means "didn't happen."
- **Confirmed benign, not a bug:** `IOPHLE remove_HLE failed for '...ReleaseCrashInfoPrev.txt'` (`Core/GameEngine/Source/Common/System/Debug.cpp`) — this is an expected-to-sometimes-fail cleanup `remove()` whose return value is deliberately discarded (no previous crash log exists on first run). Not the cause of anything downstream.
- **Traced to a real, meaningful signal: `ERROR_OUT_OF_MEMORY` thrown during the first `GameEngine::update()`.** The reset to `rom0:OSDSYS` at ~48s is `GameEngine.cpp`'s own top-level `catch(...)` around the main update loop, calling `RELEASE_CRASH` — this is the engine's *intended* crash-reporting path (writes `ReleaseCrashInfo.txt` to the user-data dir, tries a `MessageBoxW`), not a raw fault. Its log said only `Uncaught Exception in GameEngine::update` with no detail, so a temporary diagnostic was added mirroring the existing `#if defined(__ANDROID__)` exception-identifying block right next to it (`GameEngine.cpp`, `execute()`) — same technique, writing to `host:update_exception_diag.txt` instead of logcat. Result: `update threw ErrorCode: 0xdead0002`, which `Common/Errors.h`'s `ErrorCode` enum resolves to `ERROR_OUT_OF_MEMORY` (`ERROR_BASE + 1`). **This is happening with the 128 MB dev-console RAM mode active, not the retail 32 MB budget** — so some allocation during the very first update tick (almost certainly shell-map/menu asset loading, given the real `.big` files are mounted) already exceeds 128 MB, or a specific memory pool's configured cap (independent of total system RAM) is being hit. Not yet isolated to a specific pool/allocation.

**This is the real Phase 1 question, not a bug to route around.** Everything up to this point (toolchain, compile, link, the `--gc-sections`/`std::filesystem` runtime bugs) was necessary but ultimately mechanical. Hitting a genuine out-of-memory condition during the very first asset load is the actual "does this game fit" measurement the whole port plan's memory budget section was built around.

- **First real lever, confirmed working: the DMA pool table reserves ~44 MB unconditionally at startup, before any asset loads.** Traced (not assumed) via the code: `MemoryPoolFactory::createDynamicMemoryAllocator()` → `MemoryPool::init()` → `createBlob(initialAllocationCount)` runs immediately for each of the 7 fixed-size DMA pools (`GameMemoryInitDMA_GeneralsMD.inl`) at `initMemoryManager()` time — well before `GlobalData`/INI/map loading. Summed: 130k×16B + 250k×32B + 100k×64B + 80k×128B + 20k×256B + 16k×512B + 6k×1024B ≈ 44 MB, on top of the ~10.9 MB code+static footprint already measured. This table is untouched PC-era tuning (sane when hundreds of MB are free), never revisited for a console budget.
  - **Fix:** new `Core/GameEngine/Source/Common/System/GameMemoryInitDMA_PS2.inl`, wired in via a `__PS2__` branch in `GameMemoryInit.cpp` (ahead of the existing `RTS_GENERALS`/`RTS_ZEROHOUR` split, which still selects the per-type object-count `Pools_*.inl` as before — that table is small object-count tuning, not raw DMA byte budgets, and wasn't touched). Cut every pool's `initialAllocationCount` to roughly 1/8th; left `overflowAllocationCount` (the on-demand growth increment, not pre-allocated) unchanged as a safety net. Pure capacity tuning, not a behavior change — pools still grow via overflow blobs if the lower initial count isn't enough.
  - **Result: confirmed by direct measurement.** Before this fix, boot reliably hit `ERROR_OUT_OF_MEMORY` and reset to `rom0:OSDSYS` at ~48s. After, boot ran **125+ seconds** of sustained real engine execution with zero crashes, zero `IOPHLE` errors, and no `ReleaseCrashInfo.txt` — the only new log lines were unrelated PCSX2 gamepad hotplug events. This is a real, measured win, not a guess: the same `host:`-file diagnostic technique (checking for the absence of the crash-info file and exception-diag file) confirmed it, rather than assuming success from "it didn't crash immediately."
  - **Not yet done:** this was tested at PCSX2's 128 MB dev-console RAM setting, not the retail 32 MB budget — the next validation step is re-testing at 32 MB specifically, since 44 MB reclaimed is enormous progress but doesn't by itself guarantee fitting in a quarter of that headroom. Also unconfirmed: whether the reduced initial counts cause excessive overflow-blob churn once real gameplay (not just shell/menu) starts driving allocation patterns harder — worth profiling once further into a real session, not assumed fine forever.

Next lever candidates if the 32 MB retail test still comes up short (in priority order, none implemented yet): re-run the same allocation-failure diagnostic at 32MB specifically; check whether individual large assets (opened via the default whole-file `RAMFile` path rather than `File::STREAMING`) are a second contributor; then the bigger, already-planned structural work (lazy INI/template hydration, Tier 0 flat/Gouraud rendering to drop texture loads).

## Rendering bring-up (Tier 0/1 in progress)

First real gsKit bring-up in `PS2Backend.cpp`/`.h`. Two real, non-obvious bugs found and fixed via measurement, not guessing:

- **Real bug #4 (fixed): `gsKit_mode_switch(gsGlobal, GS_PERSISTENT)` was the wrong mode and caused PCSX2 itself (the host emulator, not the emulated game) to segfault.** `GS_PERSISTENT` is gsKit's "set it and forget it" pool — primitives stay queued until *manually* cleared; `gsKit_queue_exec()` does not free them. It's meant for static geometry set up once, not continuously-changing per-frame content. `GS_ONESHOT` is the double-buffered pool that swaps (implicitly clearing the just-executed half) on every `gsKit_queue_exec()` — what a real per-frame game loop actually needs. This was originally chosen (over `GS_ONESHOT`) to fix an earlier "No Image" PCSX2 GS-status bug, but that attribution was likely wrong — `GS_ONESHOT` re-tested clean once real geometry existed, so the No Image fix was probably just calling `gsKit_init_screen()`/`gsKit_mode_switch()` correctly at all. The `PERSISTENT` mistake didn't matter for Tier 0 (a single `Clear()` per frame is small enough that even the 256KB default persistent pool took 1000+ frames to fill), but once real triangle submission began (Tier 1), every triangle drawn since boot stayed queued forever, and PCSX2 (Vulkan and Software renderer both — confirmed by testing both, ruling out a renderer-specific bug) segfaulted once the cumulative total overflowed the drawbuffer. Symptoms that fit in hindsight: enlarging the buffer 16x (256KB→4MB) only delayed the crash proportionally (52→347 frames) rather than fixing it; the crash point varied run to run (52, 102, 347 frames) since it depended on cumulative triangles drawn since boot, not any single frame's content; the last-attempted triangle's data was always completely valid at the point of crash. Fixed by switching to `GS_ONESHOT` in `PS2Backend::Initialize()`. Confirmed by measurement: 228,870 real triangles across 1169 frames with zero crashes, vs. crashing every time around 3,000-63,000 triangles before.
- **Real bug #5 (fixed): `gsGlobal->PrimAlphaEnable = GS_SETTING_ON` made real geometry invisible.** After the crash fix, triangles submitted successfully with valid on-screen coordinates and solid white (`0xFFFFFFFF`) color, but nothing appeared — screen stayed black. Disabling `PrimAlphaEnable` (`GS_SETTING_OFF`) made real menu UI geometry visible immediately (a bordered list panel with divider lines, icon squares — see session screenshot). Not yet root-caused exactly *why* the GS alpha=0x80 convention assumed for "fully opaque" wasn't behaving that way; alpha blending is simply off for now since nothing currently needs it. Revisit when translucent UI/effects are actually needed.
- Real per-frame render path confirmed (via temporary instrumentation, since removed except for throttled counters): the shell/menu screen submits essentially 100% of its geometry through the *dynamic* vertex/index buffer path (`DynamicVBAccessClass`/`DynamicIBAccessClass`, captured via `Capture_Dynamic_Vertex_Data`/`Capture_Dynamic_Index_Data`), not the static `VertexBufferClass` path. `PS2Backend` implements both, keyed by separate cache maps; dynamic caches are cleared every `Begin_Scene()` since that content is only ever valid for one frame (the underlying ring buffer gets overwritten every frame by engine design).
- Software T&L implemented directly in `PS2Backend` (no VU1 yet, per the plan's own phasing) using `FVFInfoClass` (already existed in the engine, `dx8fvf.h`) for vertex-layout offsets, and the engine's existing `Matrix4x4`/`Matrix3D` classes for transform math (`Transform_Vector`, confirmed row-major/column-vector convention by reading its implementation directly rather than assuming). `D3DFVF_XYZRHW` (pre-transformed 2D UI quads) vertices use their x/y directly as screen pixels, skipping the World/View/Projection path entirely — this is the common case for menu rendering.
- Texture sampling is still out of scope (untextured flat/gouraud-shaded triangles only) — next step once this is stable.

### Texture support (same day, following the above)

Implemented in `PS2Backend`: `Register_Loaded_Texture`/`Set_Texture` mirror `BgfxBackend`'s already-working `EnsureBgfxTexture` pattern rather than inventing a new one — lock the D3D8 stub texture's real pixel data (`Peek_D3D_Texture()->LockRect()`, genuinely populated under `GGC_BGFX_STANDALONE`'s `StubD3D8Device` even though nothing is GPU-backed), convert uncompressed formats (A8R8G8B8/X8R8G8B8/R8G8B8/R5G6B5/A1R5G5B5/X1R5G5B5/A4R4G4B4/X4R4G4B4) to RGBA8 on the CPU, upload via gsKit's `gsKit_TexManager_bind`/`gsKit_prim_triangle_goraud_texture_3d`. UV is texel-space (0..Width/0..Height), not the engine's normalized [0,1] — confirmed via `gsKit_float_to_int_u/v` in gsKit's own `gsInline.h`, not assumed.

- **Real bug #6 (fixed): a 1024x1024 RGBA32 texture is exactly 4MB — the *entire* GS VRAM budget on real PS2 hardware (and PCSX2 enforces the same limit) — and hung `gsKit_TexManager_bind`'s VRAM allocator outright** (no crash, no progress, diag counters frozen solid) rather than failing gracefully. Root-caused via per-step `host:` diagnostics logged once per distinct texture (cheap, not a hot path): three small 64x64 textures bound and drew fine in sequence, the fourth (1024x1024) hung exactly at the `gsKit_TexManager_bind` call, confirmed by dimension/step logging, not guessed. Fixed with a conservative 128x128 (64KB) size cap in `EnsurePS2Texture` — textures above that are rejected and the caller falls back to untextured flat color, same as unsupported formats. This is exactly why the original PC assets are DXT-compressed: uncompressed RGBA8 was never going to fit this hardware's ~4MB video memory at real texture sizes. DXT-compressed and palettized formats are not yet decoded (same untextured fallback) — this is what's currently blocking full UI artwork (backgrounds, logos, which are typically DXT-compressed) from appearing; small uncompressed icon-sized textures already work end to end.
- Repeated the same per-triangle-logging performance mistake from earlier (see above) in the new texture-bind path and caught it the same way (FPS 30→6, removed the hot-path logging, restored). Confirmed via measurement after the fix: 852 real frames / 165,084 triangles in the same wall-clock window with zero crashes, vs. 168 frames with the hang-diagnostic logging still in place.

### DXT decompression implemented — but doesn't by itself unblock real background art

Added BC1/BC2/BC3 (DXT1/3/5) software block decompression (`DecodeDXT`/`DecodeDXTBlock` in `PS2Backend.cpp`), reusing `DXTUtils.h`'s pitch/row math (already proven correct against this exact stub-texture pipeline by `StubD3D8Device`/`BgfxBackendTextures`). Builds and runs clean, no regressions (837 frames / 161,864 triangles).

It never actually got exercised on this game screen, though, and that reveals the real constraint: per-texture diagnostics showed the only DXT-format textures here are 512x512 and 1024x1024 (the real background/menu art) — exactly what the 128x128 size cap rejects *before* format-specific decode ever runs. Decompression only shrinks *storage*; once decoded to RGBA8 for GS upload, a 512x512 texture is still 1MB (a quarter of the whole 4MB GS VRAM budget) regardless of source compression. So DXT support was necessary but not sufficient — only small already-uncompressed icons (64x64) currently render as real textures.

**Real next step to see full background art** (not done yet): downsample large source textures before upload (simple, lossy), or switch to GS's native palettized PSMT8/PSMT4 format instead of always expanding to RGBA8 (the actual PS2-appropriate answer — matches how retail PS2 titles fit textures in 4MB VRAM — but needs color quantization + CLUT management), or real VRAM texture streaming/eviction. None implemented yet.

### Downsampling implemented — closes the loop

`EnsurePS2Texture` now decodes at native resolution into a scratch buffer (main EE RAM, far more headroom than the 4MB GS budget), then repeatedly halves via a 2x2 box filter (`HalveRGBA8`) until it fits the 128x128 cap, instead of rejecting oversized textures outright. Confirmed by measurement: all 13 distinct textures this screen uses now succeed (previously 3 of them — the 512x512/1024x1024 background art — were rejected and never rendered); zero crashes, 847 frames / 164,438 triangles, no regression from the prior stable baseline.

This VRAM constraint is confirmed real, not PCSX2-specific — PCSX2 emulates the actual GS hardware's 4MB VRAM size faithfully, so the resolution-for-VRAM-fit tradeoff would be needed on real hardware too. The 32MB *retail* RAM budget (as opposed to the 128MB dev-console mode used for all testing so far) is still unverified — see the "Runtime boot debugging" section above; that remains the other open feasibility question for real hardware, separate from this GS VRAM finding.

### Real text rendering — complete working main menu achieved

The game's font path is built entirely around Windows GDI (`FontCharsClass::Initialize_GDI_Font`, real `CreateFont`/`GetGlyphOutline` calls) — nonexistent on PS2. Already solved for Android/macOS/Switch/Linux/WASM in `render2dsentence.cpp` via an `stb_truetype` fallback rasterizing against a system or embedded TTF; Switch (no system font, no reliable filesystem) already has a self-contained embedded font (`ggc_embedded_font.h`, ~180KB TTF baked into the binary). Added `__PS2__` alongside `__SWITCH__` in the two gating spots — same problem, same fix.

**Real bug #7 (fixed): the embedded-font path did an unnecessary `malloc`+`memcpy` of the ~180KB embedded TTF for every distinct font (name/size/bold) the game requested, never freed until that font's own release** — reintroduced the exact same `ERROR_OUT_OF_MEMORY` `RELEASE_CRASH` the earlier DMA-pool fix solved (confirmed via the same `host:update_exception_diag.txt` technique: `0xdead0002`), crashing at ~50s/~20 frames. Fixed with an `ownsData` flag on `GGCStbFont` — the embedded case now points directly at the static array (already has program lifetime, zero copy cost) instead of duplicating it per font instance; `Free_GDI_Font` only frees when `ownsData` is true (still true for the real-file `fopen` path other platforms use). Confirmed by measurement: 1141 frames / 232,454 triangles, zero crashes, no crash-info file written — vs. crashing every time around 20-30 frames before.

**Result**: a complete working PS2 Zero Hour main menu — real logo texture, real menu geometry, real legible text ("SOLO PLAY", "MULTIPLAYER", "LOAD", "OPTIONS", "CREDITS", "EXIT GAME"), stable 30 FPS, zero crashes. First time geometry + textures + text all work together.

Also fixed the same day: the screenshot capture script (`ps2-port/bringup-exceptions/capture_window.ps1`) used `CopyFromScreen` at screen coordinates, which grabs whatever window is topmost there regardless of ownership — it had repeatedly captured unrelated windows, once something that looked like unrelated private content from a different application. Rewrote to use `PrintWindow(hwnd, hdc, PW_RENDERFULLCONTENT)`, capturing by window handle directly; `PW_RENDERFULLCONTENT` is required for GPU-accelerated (D3D/Vulkan/OpenGL) windows or the capture renders blank. Confirmed reliable across several captures since.

**Not yet done**: a pre-baked bitmap font atlas (using the existing texture pipeline) instead of live `stb_truetype` rasterization would likely suit PS2 better — the traditional PS2-era approach, eliminates runtime rasterization cost, and a small quantized atlas would likely cost less resident memory than the outline font even post-dedup. Flagged as a good next step, not urgent since the current approach now works without crashing.

**Working diagnostic technique for this phase:** PCSX2's `HostFs` must be `true` in `PCSX2.ini` (`[EmuCore] HostFs = true`) for a booted ELF to read/write arbitrary files on the `host:` mount (not just the boot ELF itself, which PCSX2 always exposes read-only regardless of this setting). Once enabled, a program can `fopen("host:somefile.txt", "w")` and the result is immediately readable from the host side — far more reliable than screenshotting PCSX2's rendered output (this engine's `PS2Backend` doesn't submit real draws yet anyway) and avoids needing PCSX2's own native screenshot feature (Settings > Hotkeys, default F8, saved to `Documents/PCSX2/snaps`) for this kind of headless diagnostic. Use a disposable `try*` value/temporary `#if defined(__PS2__)` block writing to `host:` at the suspect call site, rebuild, boot, read the file directly — much faster than disassembly-only tracing for anything that isn't a hard crash.

### Shell-map load OOM: memory census

With the main menu working, loading the shell map's 3D content still hits `ERROR_OUT_OF_MEMORY` at ~128MB dev-console RAM. A user-directed correction ("if we run out of memory... there could be a much larger thing already taking up memory") redirected the investigation from a single failing 2MB allocation to a full census, using disposable `host:` diagnostics at every relevant allocation site plus `mallinfo()` for the authoritative total:

- `mallinfo()` at crash time: ~113-119MB in use out of a ~119MB arena (essentially no free heap left).
- `StubD3D8Device` scratch (texture/vertex/index buffer copies behind the stub D3D8 device): ~28MB ever-allocated.
- All named `MemoryPool`s combined (added `MemoryPoolFactory::dumpAllPoolsPS2()` since the engine's own `memoryPoolUsageReport` is compiled out under `RTS_BUILD_OPTION_DEBUG=OFF`): ~28MB live.
- The remainder — and the dominant contributor — was allocations too big for any DMA pool bucket, falling through to `MemoryPoolSingleBlock::rawAllocateSingleBlock()` (an individual `calloc()` per allocation): ~126-132MB *ever*-allocated across 6,154 individual calls. Logged each one to `host:ps2_rawblock_log.txt`.

**Real bug #8 (partially fixed, root cause still open): `GameMemoryInitDMA_PS2.inl`'s DMA pool table topped out at 1024 bytes, so anything bigger fell through to a raw, unpooled `calloc()`.** 82% of the 6,154 raw-block calls (5,065) were exactly one of five sizes just over that ceiling (1088/2176/4064/4352/8704 bytes). Added five new buckets to `GameMemoryInitDMA_PS2.inl` sized to catch them. First attempt silently only caught the smallest new bucket (1088) — `DynamicMemoryAllocator::init()` clamps `numSubPools` to `MAX_DYNAMICMEMORYALLOCATOR_SUBPOOLS` (8) with no warning, and the PS2 table's 12 entries (7 original + 5 new) exceeded that shared, cross-platform cap. Fixed by raising the cap to 16 under `#if defined(__PS2__)` in `Core/GameEngine/Include/Common/GameMemory.h` (other platforms' `m_pools[]` array just gets 8 unused extra slots; their own `DefaultDMA` tables are untouched and still only populate 7-8 entries).

**Result, honestly: the pool fix worked exactly as designed but did not fix the crash.** Raw-block call count dropped 90% (6,154 → 608) once all five buckets actually registered, and the five target sizes disappeared from the raw-block log entirely. But `mallinfo()` in-use stayed essentially flat across all three test runs (~118.7MB → ~119.4MB → ~119.2MB), and the engine still hits `ERROR_OUT_OF_MEMORY` during shell-map load (just at a different specific allocation each time, since fragmentation shifted). This confirms the ~119MB is real, necessary data volume — not allocator overhead or fragmentation waste — so pool-bucket tuning alone cannot close a ~9MB (or more, once retail's 32MB budget is considered instead of 128MB dev RAM) gap. **Do not re-attempt allocator/pool tuning as the fix here; the next step is identifying what data is actually 100+MB and cutting or streaming it.**

With the fragmentation noise cleared, the remaining raw-block log (608 entries, `host:ps2_rawblock_log.txt`) is a much cleaner signal for that: the single largest identified contributor is 120 allocations of exactly 349,680 bytes (~42MB alone), followed by 108×17,408 (~1.9MB), 81×87,536 (~7MB), 40×87,408 (~3.5MB), 15×174,904 (~2.6MB). These round, per-instance sizes look like real per-object/per-template data, worth tracing next.

**Attribution attempt, and a real methodology correction.** `__builtin_return_address(N>0)` reliably returns `0x0` on this MIPS/R5900 target regardless of `-fno-omit-frame-pointer` (GCC's own manual notes frame-chain walking isn't supported on every architecture; MIPS is one). The literal-tag call-site mechanism (`PASS_LITERALSTRING_ARG2`) is also unavailable, gated behind `MEMORYPOOL_DEBUG`/`RTS_DEBUG`. Worked around with a poor-man's stack scan: read raw words upward from `$sp` and log any that land inside the ELF's `.text` VMA range (from `readelf -S`); MIPS still spills `$ra` to the stack on any nested call, so real return addresses show up at consistent offsets across repeated calls even without a maintained frame-pointer chain. Ran it for the 349,680-byte size specifically (all 120 calls) and `addr2line`'d the ~59 unique candidate addresses against the unstripped ELF. Result: a clean, consistent chain — `WW3DAssetManager::Get_Texture` → `TextureLoadTaskClass::Load_Compressed_Mipmap`/`Begin_Compressed_Load` → `DDSFileClass::DDSFileClass` (`ddsfile.cpp:298`, `DDSMemory = new unsigned char[size]`, the full compressed DDS mip-chain minus header) → `StubD3D8Device::CreateTexture`.

That looked like a smoking gun, but checking `DDSFileClass`'s lifetime disproved it: every call site declares it as a function-local stack variable (`DDSFileClass dds_file(...)` in `textureloader.cpp`), destroyed — freeing `DDSMemory` via `delete[]` in `~DDSFileClass()` — before the next texture loads. One at a time, not concurrently held. This exposed a flaw in the whole investigation up to this point: **`g_ps2RawBlockTotalBytes`, `s_ps2PoolBytesAllocated`, and `s_ps2ScratchTotalBytes` are all monotonic "ever allocated" counters that never decrement on free** — they measure cumulative churn, not what's actually resident when the crash happens. Only `mallinfo()`'s in-use figure and the named-pool `dumpAllPoolsPS2()` breakdown (which uses `getUsedBlockCount()`, a true live count) were trustworthy live numbers.

**Fix: added real live (alloc-minus-free) tracking for the raw-block path.** First attempt used `std::unordered_map<void*, Int>` as the alloc→size side-table — this immediately hung boot solid (confirmed: zero forward progress for 90+ seconds, only the very first raw-block allocation ever logged). Root cause: this file overrides the global `operator new`/`new[]` to route through `DynamicMemoryAllocator`, so `std::unordered_map`'s own first-use bucket allocation recurses straight back into the same `allocateBytesDoNotZeroImplementation` this code was instrumenting — the exact reentrancy trap `StubD3D8Device.cpp`'s `AllocScratch` already hit and documented (fixed there with `std::calloc` bypassing the pool). Fixed identically in principle: replaced the map with a plain fixed-size array (`s_ps2RawBlockLivePtrs`/`s_ps2RawBlockLiveSizes`, capacity 16384, linear-scan insert/lookup/erase) — no dynamic allocation at all, so it can't recurse. Applied the same fix to the size-histogram dump at OOM-halt time (doubly important there, since the heap is already known exhausted at that point).

**Real live picture (confirmed by measurement, same run):** `mallinfo()` in-use ~115.6MB. Named pools live: ~37.1MB (`dumpAllPoolsPS2`). Raw blocks live: only **~6.7MB across 138 blocks** (`host:ps2_rawblock_live_breakdown.txt`) — confirming the ~94MB "ever allocated" raw-block figure was almost entirely churn, not residency; the 349,680-byte size doesn't even appear in the live breakdown anymore. `StubD3D8Device` scratch: ~27.3MB *ever*-allocated (not yet live-tracked, so this is an upper bound). Even in the worst case (100% of scratch still live), 37.1 + 6.7 + 27.3 ≈ 71MB — **leaving ~44MB of live memory at OOM time still completely unattributed to anything `GameMemory.cpp` or `StubD3D8Device.cpp` can see.** This means a meaningful chunk of real, resident memory is going through some other allocation path entirely — plain `malloc`/`new` calls that bypass `DynamicMemoryAllocator` (candidates: `ArchiveFileSystem`/zlib `.big`-extraction buffers, SDL3 internals, or other third-party code) — not yet identified.

**Attempted, real-negative-result optimization: releasing StubD3D8Texture's native-resolution scratch after PS2Backend's one-time downsampled-copy read.** `PS2Backend::EnsurePS2Texture` caches a downsampled RGBA8 copy of every texture on first draw and never reads the underlying `StubD3D8Texture` again (confirmed: `m_textureCache.find(tex)` short-circuits on every subsequent call before reaching any `LockRect`). That makes the D3D8-level native-resolution scratch (potentially several MB per texture with mips, ~27MB total ever-allocated across all textures) dead weight for the rest of the texture's lifetime — in principle. Added `StubD3D8Texture::ReleaseNativeScratchIfManagedPS2()` (gated `D3DPOOL_MANAGED` only, matching real D3D8 semantics for driver-regeneratable content, and matching the confirmed pool used by all real file/DDS-loaded textures), called from `PS2Backend.cpp` right after the downsampled copy is cached. **Result: a real no-op, confirmed by measurement (`mallinfo()` byte-identical before/after, same crash at the same allocation size).** Root cause: `StubD3D8Surface`'s "borrowing constructor" captures its own raw copy of the scratch pointer at construction time (independent of the texture's `unique_ptr`, so freeing the texture's copy out from under an existing surface would dangle it — a real correctness risk this method correctly refuses to take). `TextureBaseClass::Load_Locked_Surface()` (`texture.cpp:317`) calls `GetSurfaceLevel(0, ...)` as a normal part of the shared texture-loading pipeline, which runs for every texture well before PS2Backend ever sees it — so a surface already exists for level 0 by the time `EnsurePS2Texture` would try to release it, and the safety check correctly bails out every time. The scratch is structurally load-bearing under the current `StubD3D8Surface` aliasing design (shared across Android/macOS/iOS/WASM too, not PS2-specific architecture), not free real estate — closing this would require restructuring `StubD3D8Surface` to reference back through its owning texture+level instead of holding a raw pointer, a real cross-platform-shared change, not a quick PS2-only win. The code is kept (harmless no-op, documents the investigation) rather than reverted.

**Where this leaves the 128MB/32MB budget question.** With real live numbers now in hand: ~37MB named pools + ~6.7MB raw blocks + ~27MB scratch (upper bound, likely mostly genuinely live given the above) + ~44MB still-unattributed ≈ 115MB matches `mallinfo()`. None of the "quick win" avenues tried so far (pool bucket tuning, scratch early-release) recovered meaningful memory, because neither was actually waste — both were real, structurally-necessary data. Closing the gap to fit 128MB (need ~10-15MB headroom for safety) or especially 32MB retail (need to cut over 70%) most likely requires either: (1) finding and fixing the ~44MB unattributed source, which could be genuine waste (e.g. an oversized read buffer, a decompression scratch never freed) rather than necessary data — worth live-tracking `StubD3D8Device` scratch the same way raw blocks were, and/or a broader `malloc`/`new` interposition to catch allocations outside `GameMemory.cpp` entirely; or (2) real content/architecture cuts (fewer resident `ThingTemplate`s, lazy-loaded pools, smaller texture budget) rather than allocator-level fixes, since the ~37MB of named-pool data and the texture scratch both turned out to be real, structurally necessary memory once measured properly rather than allocator overhead.

## Memory budget, continued: the real waste was hiding in blob over-provisioning

Added a linker-level `-Wl,--wrap=malloc/calloc/realloc/free` around the real allocator (`Core/GameEngine/Source/Common/System/PS2MallocWrap.cpp`, wired as a `target_link_options` on `z_generals` specifically — **not** `CMAKE_EXE_LINKER_FLAGS`, since that also applies to CMake's own internal compiler-sanity-check test executables during configure, which don't link `PS2MallocWrap.cpp`'s `__wrap_*` definitions and fail to link with "undefined reference to `__wrap_malloc`", confirmed the hard way first). Same fixed-array live-tracking technique as the raw-block fix (a `std::unordered_map` side-table here hits the identical reentrancy trap: its own first-use allocation calls `malloc`, recursing into `__wrap_malloc` and hanging boot solid — confirmed by direct measurement, fixed with plain arrays). This catches every allocation in the program, including ones `GameMemory.cpp`/`StubD3D8Device.cpp` already track (so its live total is expected to roughly match `mallinfo()`, not just show the gap) — the per-size histogram is what actually localizes new sources.

`__builtin_return_address(N>0)` is still 0x0 on this MIPS target (confirmed again, not architecture-specific to the first investigation). Reused the poor-man's-stack-scan technique (read raw `$sp`-relative words, keep any landing in the `.text` VMA range from `readelf -S`, `addr2line` the consistent-across-repeats ones) against the largest unexplained live sizes. First three targets (32768, 2097152, 72096, 17420 bytes) all resolved to memory already counted elsewhere — `StubD3D8Device::CreateTexture`/`CreateVertexBuffer`/`AllocScratch` (already in the scratch total) and `INI::initFromINIMulti`'s `ModelConditionInfo` vector growth (exactly the already-known 17408-byte raw-block entry; the 12-byte difference is `MemoryPoolSingleBlock::calcRawBlockSize`'s header) — real negative results, not wasted effort, since they ruled out those paths.

**The fourth target was the real find.** A single 10,528,000-byte (10.5MB) live allocation resolved cleanly to `WW3DAssetManager::Load_3D_Assets` → `Load_Prototype` → `MeshClass`'s memory pool creating its very first blob (`MemoryPool::addToList` → `MemoryPoolBlob::initBlob` → `sysAllocateDoNotZero`). Cross-referencing the pool breakdown: `MeshClass allocSize=740 used=1028 total=14000` — the pool's PC-era `initialAllocationCount` (14000, from the shared `GameMemoryInitPools_GeneralsMD.inl`) reserves one giant upfront blob sized for 14,000 mesh objects, but the shell map only ever instantiates 1,028 of them. **This exposed a real blind spot in the whole session's live-tracking methodology up to this point:** `MemoryPoolFactory::dumpAllPoolsPS2()` reports `usedBlockCount × allocSize` as a pool's "live bytes" — which is correct for logical usage, but doesn't count a blob's actual *reserved* size when a pool is heavily over-provisioned. `mallinfo()`/the malloc-wrap tool both correctly count the full reserved blob as live (it's real, resident memory from the OS's perspective), but no per-pool report had been checking blob-reserved-vs-block-used before this. Auditing every pool in the same breakdown for `(total - used) × allocSize` found this pattern was widespread, not just `MeshClass`: `MeshMatDescClass`/`MeshModelClass`/`VertexMaterialClass`/`MaterialInfoClass`/`DynD3DMATERIAL8` all reserve 6000-8192 slots for single-digit usage; `MotionChannelClass`/`ShareBufferClass` reserve 16384/32768 for 16/54 used. **Summed across every named pool (including the PS2-specific DMA buckets, some of which had already grown via one or more overflow blobs that were themselves mostly empty, e.g. `dmaPool_128`: 10000 initial + one 10000-block overflow = 20000 total for 12331 used), total reserved-but-unused waste measured at ~27.8MB** — nearly the entire "unattributed" gap from the earlier census.

**Fix, two parts, both real measured wins:**
1. `Core/GameEngine/Source/Common/System/GameMemoryInit.cpp`: added a small `PS2PoolOverrides[]` table applied inside `userMemoryAdjustPoolSize()`, right-sizing `initialAllocationCount` for the dozen worst-offending per-class pools (`MeshClass` 14000→1500, `MotionChannelClass` 16384→250, `ShareBufferClass` 32768→250, etc., all with real headroom above measured shell-map usage since actual matches reference more distinct meshes/materials than the menu does). Deliberately a small override layered on the shared `PoolSizes[]` table, not a forked copy of the whole multi-hundred-entry table — a fork would silently drift out of sync with balance changes made for other platforms.
2. `GameMemoryInitDMA_PS2.inl`: raised `initialCount` for the buckets that had already grown past it (128/512/1024/1088/2176/4064/4352/8704) to cover measured shell-map usage without needing overflow growth, and shrank their `overflowCount` so a real match that does need more doesn't over-commit another huge blob the same way.

**Result, confirmed by measurement:** total pool waste (same `(total-used)×allocSize` audit) dropped from ~27.8MB to ~9.1MB. But `mallinfo()` in-use barely moved (~115.6MB → ~115.2MB, only ~433KB less) — because the freed headroom let the shell-map load process run *further* before hitting the same OOM ceiling, not further than the ceiling. Evidence this is real progress, not a wash: `StubD3D8Device` scratch grew from ~27.3MB/~26.8MB (ever/live) to ~37.9MB/~33.3MB, raw blocks grew from ~6.7MB live/138 slots to ~19.9MB live/406 slots, and the named-pool breakdown now shows a *different* set of objects (more textures, more map objects, `Parameter`/`MapObject`/`UVBufferClass` entries that weren't visible before) — the game genuinely loaded more content with the freed-up room before running out again. **The underlying conclusion stands: this class of fix (allocator/pool right-sizing) recovers real, measurable memory, but shell-map loading's total demand still exceeds 128MB even after it — the next lever needs to reduce what actually gets *loaded*, not how efficiently it's packed.**

## Texture load-time reduction: real fix, real progress, still not enough alone

Texture scratch was the fastest-growing tracked category after the pool fix, so the next target was load-time texture resolution: PS2 loads every DDS texture at full native resolution into `StubD3D8Device` scratch, only for `PS2Backend::EnsurePS2Texture` to immediately halve it down to a 128×128 GS-VRAM cap moments later. `GlobalData::m_textureReductionFactor` → `WW3D::Get_Texture_Reduction()` → `DDSFileClass`'s mip-skip loop is a real, already-shipped mechanism (confirmed by reading the code end-to-end) that would make the loader skip reading the largest mips in the first place, avoiding that waste with zero on-screen change (final resolution is already capped at 128 either way).

**First attempt (real negative result): calling `WW3D::Set_Texture_Reduction(3, 32)` directly at the end of `WW3D::Init()`.** Confirmed by measurement to be a complete no-op — `mallinfo()` came back byte-identical. Root cause: `GameLODManager::applyStaticLODLevel()` runs later in boot and unconditionally recomputes the reduction via `getLevelTextureReduction(getRecommendedTextureLODLevel())`, then calls `TheGameClient->setTextureLOD()`, silently overwriting the early call. `getRecommendedTextureLODLevel()` returns `STATIC_GAME_LOD_LOW` whenever `!m_memPassed` (true on PS2, confirmed), but `StaticGameLODInfo`'s per-level `m_textureReduction` field only gets a non-zero value from INI-parsed `GameLOD.ini` data for every level except the hardcoded `VERY_HIGH` default — in this build/test environment that resolved to 0.

**Fix: moved the override into `GameLODManager::applyStaticLODLevel()` itself** (`GeneralsMD/Code/GameEngine/Source/Common/GameLOD.cpp`), as a floor (`if (requestedTextureReduction < 3) requestedTextureReduction = 3;`) applied under `__PS2__` right where `requestedTextureReduction` is computed — the same function that was overwriting the earlier attempt, so this is guaranteed to be the actual last writer before `setTextureLOD()` applies it. A floor, not a hard override, so real INI data (if ever present and more aggressive) still wins.

**Result, confirmed by measurement: a real, meaningful win, not a full fix.** `StubD3D8Device` scratch dropped from ~37.9MB/~33.3MB (ever/live) to ~30.7MB/~26.0MB with reduction=3 — about 7.3MB live recovered. The OOM crash point moved dramatically closer to succeeding: the failing allocation request shrank from 3,145,784 bytes down to 45,056 bytes, meaning the shell-map load got much further before running out. Tried reduction=4 (more aggressive) as a follow-up: scratch dropped a little further (~24.9MB live) but the crash point actually landed *further* from success (440,000-byte failing request, worse than reduction=3's 45,056) — not a clean monotonic improvement, so reduction=3 was kept as the better-measured value; the exact optimal value is a small tuning search, not assumed.

**Why the crash persists even this close:** `mallinfo()`'s `arena` (total heap the OS has handed out) tops out around ~117-118MB across these runs, not the full 128MB PCSX2 provides — and `fordblks` (free-within-arena) stays tiny (22-33KB), meaning the heap genuinely can't grow further, not that it's fragmented. Checked `ps2sdk/ee/startup/linkfile`: `PROVIDE(_heap_size = -1)` confirms no artificial ps2sdk-imposed heap cap — the ~10MB gap between 128MB and the observed ~118MB ceiling lines up with the already-measured **~12.6MB static ELF footprint** (`.text` 10.2MB + `.data`/`.rodata`/`.bss` ~2.4MB, from `readelf -S`) plus some fixed BIOS/kernel/IOP reservation, not a new mystery. This directly confirms the `-O2`→`-Os` code-size lever (not yet attempted) as the next highest-value target: every byte trimmed from the static footprint directly grows the usable heap ceiling by the same amount, on top of the real bytes already recovered from pool right-sizing and texture reduction.

**Current state:** both fixes (pool right-sizing, texture reduction=3) are applied and kept (real, measured, no-degradation wins — final rendered quality is unchanged in both cases). The shell-map OOM is not yet resolved, but the failing allocation request has shrunk by roughly 65x (3.1MB → 45KB) across this session's fixes, and the remaining gap now points clearly at the static code-size footprint as the next lever, not another allocator/content investigation.

## `-O2` → `-Os`: real, measured, but confirms demand keeps growing with headroom

Switched `EE_FLAGS` in `cmake/toolchains/ps2.cmake` from `-O2` to `-Os` (pure code-size change, no semantic difference). Confirmed via `readelf -S`: `.text` dropped from 10.22MB to 7.68MB (~2.55MB saved, ~25%), total static footprint (`.text`+`.data`+`.rodata`+`.bss`) from ~12.6MB to ~10.0MB.

**Result, confirmed by measurement, reproducible: a real win, cleanly matching prediction.** `mallinfo()`'s heap arena grew from ~118.2MB to ~120.7MB — almost exactly the 2.55MB `.text` savings, confirming the earlier hypothesis that the ~118MB-not-128MB ceiling was the static footprint eating into the fixed 128MB total, not a separate bug. But the failing allocation request grew too, from 45,056 bytes to 1,340,000 bytes — **worse** by the "how close to success" measure, even though the underlying heap genuinely got bigger. This is the same non-monotonic pattern the pool-fix and texture-reduction-4 attempts already showed: freed headroom doesn't shrink the next failure, it lets loading progress further into content that was previously unreachable, and that content's own demand can be larger than what was just freed. **Taken together across all of this session's fixes, this is a real, sobering data point: shell-map loading's total memory demand appears to keep growing roughly in step with however much headroom becomes available, not asymptotically approaching some fixed total just above 128MB.** This doesn't mean the fixes were wrong (each is real, verified, zero-degradation, and worth keeping — a smaller footprint and less waste is strictly better regardless), but it means "give it a few more MB and it'll fit" is not a safe assumption anymore; the actual total demand for a fully-loaded shell map has not yet been observed and could be substantially larger than 128MB.

**Also this session: automatic ELF stripping wired into the build itself** (`GeneralsMD/Code/Main/CMakeLists.txt`, `add_custom_command(TARGET z_generals POST_BUILD ...)`, PS2-only, using an explicit `CMAKE_STRIP` path in `cmake/toolchains/ps2.cmake` rather than relying on auto-detection, matching the existing `CMAKE_AR`/`CMAKE_RANLIB` explicit-path pattern this toolchain already needs). Produces `GeneralsMD/Release/generalszh_stripped.elf` automatically on every build — no more manual `mips64r5900el-ps2-elf-strip.exe` invocation. **Gotcha hit while wiring this up (fixed):** the first version followed whichever Ninja Multi-Config config actually got built (`Debug`, per this whole session's established build practice), but the retail `.big` game-data files have only ever lived in the `Release/` folder — booting the `Debug/`-located stripped ELF without them produced a totally different failure (PCSX2's own BIOS memory-card browser screen, not the game, confirmed via the FPS counter showing free-running ~2294-2325fps/~3800% speed rather than the game's real ~30fps/100%). Fixed by hardcoding the strip destination to `${CMAKE_BINARY_DIR}/GeneralsMD/Release/` regardless of which config's object files fed it, rather than deriving the path from the built config.

## Running-match RAM cut pass (skirmish, 2026-07-13)

Context: with a test hook that boots straight into a 1-human/no-AI skirmish on `empty.map` (temporary, in `GameEngine::init` under `__PS2__`), the match hit a hard OOM at `mallinfo` arena ~120MB (128MB dev-console budget). This pass targeted the running-match footprint. Baseline census at the OOM point: GameMemory pools ~53MB reserved (42MB live: dmaPools 32 + named 10), StubD3D8 texture scratch ~24MB live, gsKit oneshot drawbuffer ~20MB, raw one-off blocks ~13MB live.

**Real bug found + fixed along the way — gsKit oneshot drawbuffer overflow (the "hang").** The oneshot drawbuffer (`Os_AllocSize`) is host EE RAM that accumulates one frame's GIF packets, and is **double-buffered** (`pool[2]`), so the requested size is charged twice. gsKit does **not** bounds-check writes into it: once a single frame's primitives exceed the buffer, `gsKit_prim_triangle_*`/`TexManager_bind` write past the end and silently corrupt adjacent heap — which manifests as an apparent "hang" at a point that varies run to run. Confirmed directly by logging `pool_max[dbuf] - pool_cur`, which went negative and grew steadily more negative (~144 bytes/triangle). A real skirmish frame needs far more primitives than the shell map's decorative background ever did. This is also why an earlier per-texture VRAM-eviction attempt had zero effect — wrong resource entirely (EE RAM, not GS VRAM).

**Cut #1 — gsKit oneshot buffer 10MB→1MB per buffer, with mid-frame flush (~-16MB, and fixes the corruption).** Rather than size the buffer for the worst-case frame (10MB alone pushed us OOM; the double-buffering charges it twice), keep it small and flush mid-frame when it approaches full. `PS2Backend::MaybeFlushOneshot()` (called from `DrawCapturedTriangle`, between whole primitives so a GIF packet is never split) checks remaining pool space and calls `gsKit_queue_exec` when under 128KB — kicking the accumulated packets' DMA and letting the double-buffered pool reset, without a `sync_flip` (the display framebuffer only swaps at `End_Scene`). This is the standard PS2 pattern for scenes larger than one buffer and makes the required size independent of scene complexity. Since our code already calls `gsKit_TexManager_bind` before every textured triangle, no texture-rebind handling is needed across a flush. Confirmed by boot test: the drawing environment survives the mid-frame kick, `oneshotRemaining` stays healthy positive, and the match now pushes 78+ frames / 480K+ triangles where it previously froze at ~94K binds.

**Cut #2 — lazy StubD3D8 owning-surface backing store (`__PS2__`-guarded, ~-2 to -4.6MB).** The D3D8-mirror surfaces (backbuffer, depth-stencil, `CreateRenderTarget`/`CreateImageSurface` results) exist only to give D3D8-shaped engine code non-null handles; their pixels are never read on PS2 (gsKit does all real output). A 640×448 A8R8G8B8 surface is 1.15MB; the census showed four live (~4.6MB). Deferred the `AllocScratch` from the owning ctor to the first `LockRect` (via an `m_ownsScratch` flag; borrowing surfaces are unaffected), so surfaces that are never locked never allocate. Guarded to PS2 so every other bgfx-standalone platform keeps eager allocation byte-for-byte.

**Measured result: arena 107.7MB at End_Scene #60 in a live skirmish, reaching frame 107+ with no OOM** (down from OOM-at-120MB; the match now actually loads and runs). StubD3D8 scratch live dropped 24MB→22.1MB (some surfaces still get locked). The freed headroom is partly reabsorbed because loading now progresses *further* into content that was previously unreachable — the expected non-monotonic behavior.

**Biggest remaining lever: StubD3D8 texture scratch (~22MB live).** `ReleaseNativeScratchIfManagedPS2` only frees `D3DPOOL_MANAGED` textures that have actually been drawn (called from `EnsurePS2Texture` at draw time); the 22MB is a mix of loaded-but-not-yet-drawn textures, non-MANAGED buffers, and vertex/index-buffer scratch. Cutting it (16-bit decode scratch, DXT-in-RAM, or load-time release restructuring) is a larger change with a visual-quality or architectural tradeoff — deferred. Structural levers (lazy ThingTemplate hydration for the 32MB dmaPools; zero-copy static VB/IB capture to kill the PS2Backend/StubD3D8/engine triple-storage) remain the path to the 32MB retail budget and to absorbing real sound+AI.

**Note:** these two cuts are permanent correctness/efficiency fixes valid with sound+AI enabled, not config-specific hiding. FPS in the heavy skirmish frame is currently <1 (raw 480K-triangle volume + per-frame flushes) — a separate perf concern, not addressed here.
